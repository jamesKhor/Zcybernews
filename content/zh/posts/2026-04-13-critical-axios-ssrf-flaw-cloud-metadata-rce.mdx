## 执行摘要
广泛使用的 Axios HTTP 客户端库中存在一个关键的服务器端请求伪造（SSRF）漏洞，使攻击者能够绕过安全控制，对内部服务（包括云元数据端点）发起未经授权的请求。根据研究员 Jason Saayman 的说法，这个漏洞允许泄露敏感的云实例元数据，这些元数据可以被用来窃取凭证、横向移动和实现远程代码执行，对部署在 AWS、Google Cloud 和 Azure 等云环境中的应用程序构成严重风险。

## 技术分析
该漏洞源于 Axios 的适配器层对 URL 输入处理不当。当 Axios 被配置为使用特定适配器时，例如 Node.js 中的 HTTP/HTTPS 适配器，它未能充分验证或限制请求可以针对的网络端点。这允许控制请求 URL 参数的攻击者（例如，通过用户提供的输入字段）强制应用程序服务器向内部、不可路由的地址发起 HTTP 请求，包括属于云平台内部元数据服务的那些地址。

主要风险是访问云实例元数据服务（例如，在 AWS 上为 `http://169.254.169.254/`）。该服务包含临时安全凭证、IAM 角色信息和其他敏感配置数据。通过利用这个 SSRF 漏洞，攻击者可以检索这些凭证，有效地假定云实例附加的 IAM 角色的权限。如果角色拥有过多的权限，影响会加剧，使得攻击者能够执行诸如创建新的计算资源、访问存储桶或破坏云环境中的其他服务等操作，最终导致实例上的远程代码执行。

## 入侵指标
目前未识别出任何入侵指标。

## 战术、技术与程序
核心技术是 **服务器端请求伪造（T1190 - 利用面向公众的应用程序）**。攻击者首先识别出一个使用 Axios 根据用户输入发起请求的应用程序端点。然后，他们提交一个指向云元数据端点或其他内部服务的恶意 URL。成功的利用遵循一个清晰的链条：最初的 SSRF 访问元数据服务，从元数据中窃取凭证，然后使用这些凭证在云环境中进行 **有效账户（T1078）** 以执行未经授权的操作，导致 **远程服务（T1021）** 和 **远程代码执行（T1203）**。

## 威胁行为者背景
任何具有对易受攻击应用程序端点网络访问权限的威胁行为者都可以利用这个漏洞。这包括扫描互联网上暴露服务的机会主义攻击者，以及可能将这个漏洞纳入更广泛攻击链的更有针对性行为者。利用的简单性，尤其是现在有了公开的 proof-of-concept 代码，降低了进入的门槛。主要动机是在云环境中获得初始访问权限和权限提升，以实现数据盗窃、加密挖掘或建立持久立足点。

## 缓解措施与建议
最直接的缓解措施是升级 Axios 到修补过的版本。开发者和安全团队应该审计他们的代码库，查找用户输入流入 Axios 请求 URL 而没有严格验证的情况。对所有用户提供的 URL 实施允许列表验证，拒绝任何对内部 IP 地址范围的请求（例如，`127.0.0.0/8`，`10.0.0.0/8`，`169.254.0.0/16`，`192.168.0.0/16`）。对于云部署，使用最小权限原则限制 IAM 角色和实例配置文件的权限。此外，如果可能，配置云元数据服务以要求 `X-Forwarded-For` 头部或使用最新版本的元数据服务，这些服务强制使用 HTTP 令牌，可以阻止简单的 SSRF 尝试。
