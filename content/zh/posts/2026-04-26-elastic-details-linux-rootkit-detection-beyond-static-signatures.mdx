## 执行摘要

Elastic Security Labs 发布了其 Linux 根工具检测工程系列的第二部分，重点关注基于静态签名检测的局限性和行为监控识别高级根工具的必要性。这项研究于 2026 年 4 月 26 日发布，展示了现代 Linux 根工具——特别是那些在内核内存中运行或使用无文件技术的根工具——通常通过修改运行时行为而不是留下持久文件工件来规避静态扫描。Elastic 提倡采用监控系统调用钩子、进程执行流和内核模块加载模式的检测策略。

## 技术分析

Elastic Security Labs 的分析强调，静态检测方法——哈希匹配、YARA 规则和文件系统扫描——无法对抗仅加载到内存中或利用合法内核模块进行恶意目的的根工具。研究人员详细说明了根工具如何钩住系统调用，如 `sys_open` 或 `sys_read`，以隐藏进程、文件和网络连接，而不将任何恶意二进制文件写入磁盘。Elastic 的检测工程方法依赖于观察异常行为模式：意外的系统调用重定向、隐藏的内核模块和 `/proc` 列表与实际进程活动之间的差异。

一个关键发现是，许多 Linux 根工具现在使用 **LD_PRELOAD** 或通过 `insmod` 进行 **内核模块加载** 来在运行时拦截系统调用。这些技术允许根工具在积极操纵操作系统对其自身视图的同时，对文件完整性检查保持不可见。Elastic 的行为检测将多个内核源的事件相关联——系统调用表、进程创建和网络套接字监控——以标记表明根工具存在的不一致性。

报告还检查了诸如 **DKOM（直接内核对象操作）** 等规避策略，其中根工具在内存中更改内核数据结构以隐藏进程或提升权限。Elastic 指出，检测 DKOM 需要将内核对象状态与从系统调用输出派生的预期值进行比较，这是一种静态扫描仪无法复制的技术。

## 缓解措施与建议

Elastic 建议防御者部署能够监控内核级事件的端点检测和响应（EDR）解决方案，而不是仅依赖基于文件的扫描。具体的缓解措施包括启用 Linux 安全模块（LSMs）如 SELinux 或 AppArmor 以限制内核模块加载，审计系统调用表以查找未经授权的修改，并使用基于 eBPF 的监控来检查运行时行为。组织还应实施对 `insmod` 和 `modprobe` 使用的严格控制，以防止未经授权的内核模块安装。研究强调，没有单一的检测方法足够；结合行为分析和定期内存取证的分层方法，为抵御复杂的 Linux 根工具提供了最佳防御。
