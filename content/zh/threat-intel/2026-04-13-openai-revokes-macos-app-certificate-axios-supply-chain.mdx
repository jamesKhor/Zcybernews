## 执行摘要
OpenAI 在软件供应链攻击引入恶意版本的流行 `axios` HTTP 客户端库到其构建过程后，撤销了其官方 macOS 桌面应用程序的开发人员证书。该事件发生在 2026 年 3 月 31 日，当时一个受威胁的 GitHub Actions 工作流程下载了受污染的库。OpenAI 表示，没有用户数据或内部系统受到威胁，但公司撤销了应用程序的证书，作为预防措施，以防止其受信任的开发人员身份的潜在滥用。

## 技术分析
事件的核心是 GitHub Actions 工作流程的威胁，该工作流程负责构建和代码签名 OpenAI 的 macOS 应用程序。代码签名使用苹果的数字证书来验证应用程序的真实性和完整性。根据 OpenAI 的说法，这个自动化流程被配置为在构建过程中下载依赖项，包括 `axios` 库。在 3 月 31 日，工作流程获取了一个恶意版本的 `axios`。恶意代码的确切性质及其预期的有效载荷由 OpenAI 保留，并未在可用的来源中详细说明。

直接后果是产生了一个 macOS 应用程序二进制文件，虽然仍然用 OpenAI 的合法证书功能性签名，但包含未经授权的代码。OpenAI 的主要遏制行动是撤销用于签名受影响构建的特定开发人员证书。这一行动使应用程序的签名在所有系统上无效，导致 macOS 在用户尝试安装或运行它时将其标记为不可信。公司强调，威胁被隔离到构建管道，并未扩展到其生产服务、模型或用户数据。

## 入侵指标
目前没有识别出任何入侵指标。OpenAI 尚未公开发布与特定恶意 `axios` 包或受威胁的应用程序构建相关的文件哈希、版本号或网络指标。

## 战术、技术与程序
根据披露的事件序列，威胁行为者的 TTPs 可能涉及：
*   **T1195.002: 供应链威胁 - 软件依赖项：** 主要攻击向量是污染公共软件包（`axios`）或威胁其分发机制（例如，npm 注册表帐户、仓库）。
*   **T1552.001: 未安全凭证 - 文件/构建中的凭证：** 攻击者可能试图窃取嵌入在 GitHub Actions 环境中的代码签名证书或 API 密钥，尽管 OpenAI 的证书撤销表明这是一种防御性预防措施，而不是确认的盗窃。
*   **T1588.001: 获取能力 - 恶意软件：** 行为者创建并分发了一个伪装成合法依赖项的恶意软件库。
*   **滥用自动化工作流程：** 攻击利用了对自动化 CI/CD 管道的信任，其中下载和集成外部依赖项是在没有手动审查每个构建的情况下发生的。

## 威胁行为者背景
可用的源材料没有将此事件归因于已知的威胁行为者或团体。通过其软件供应链针对高调的 AI 公司表明，行为者可能是出于财务动机或旨在获得初始访问权限以进行随后的间谍活动或数据盗窃。然而，缺乏关于恶意有效载荷的详细信息以及事件明显被遏制在构建系统中，使得根据当前信息无法确定归属。

## 缓解措施与建议
维护类似软件分发管道的组织应考虑以下行动：
*   **实施依赖项审查：** 对软件依赖项实施严格控制。使用工具锁定依赖项版本和哈希（例如，npm 的 `package-lock.json`，`npm shrinkwrap`）并验证完整性校验和。考虑为所有构建依赖项使用一个私有的、策划的工件仓库。
*   **加强 CI/CD 安全：** 将 GitHub Actions 工作流程限制为最小必要权限。使用 OpenID Connect (OIDC) 代替长期秘密的云凭证。实施对工作流程文件和关键构建脚本更改的强制性代码审查。
*   **采用二进制授权：** 对于编译应用程序，使用二进制授权策略，要求在构建符合代码签名条件之前进行额外验证（例如，SLSA 出处，代码审查签名）。这在构建和签名步骤之间创建了分离。
*   **准备证书撤销计划：** 如果怀疑威胁，有一个明确的响应计划，用于快速撤销和替换代码签名证书，包括客户通知程序。
*   **监控证书滥用：** 在事件之后，监控代码签名证书透明度日志和安全源，以查找任何未经授权的使用被撤销的证书。
