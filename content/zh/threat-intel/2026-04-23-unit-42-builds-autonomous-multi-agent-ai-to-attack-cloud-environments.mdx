## 执行摘要

Palo Alto Networks 的 Unit 42 研究团队开发并展示了一个能够自主攻击云环境的多代理人工智能系统，该系统能够将多个攻击步骤串联起来——从初步侦察到权限提升和数据外泄——无需人为干预。这项研究于 2026 年 4 月 23 日发表，提供了一个具体的证明概念，即大型语言模型（LLM）驱动的代理可以协调利用云配置错误和安全漏洞，在现实场景中进行攻击。Unit 42 强调，这项工作旨在帮助防御者理解和准备应对新兴的 AI 驱动威胁，而不是促进恶意使用。

## 技术分析

Unit 42 的系统使用多个专门的 LLM 代理，每个代理在攻击链中扮演不同的角色。架构包括一个规划代理，它将高级目标分解为子任务，一个执行代理，运行云 CLI 命令和 API 调用，以及一个评估输出并调整计划的批评代理。代理通过共享内存缓冲区进行通信，使它们能够适应变化的云环境并从故障中恢复。

在对模拟和受控云环境的测试中，多代理系统成功地：
- 发现了公开暴露的存储桶和数据库
- 利用配置错误的 Identity 和 Access Management (IAM) 角色提升权限
- 在虚拟私有云 (VPC) 边界之间横向移动
- 将合成的敏感数据外泄到攻击者控制的端点

研究人员指出，系统的效率在很大程度上取决于底层 LLM 的质量和代理提示的具体性。当提供“破坏云环境”等宽泛目标时，系统会从预定义的工具包中自动选择适当的工具和技术，包括 AWS CLI、`jq` 和 `curl`。Unit 42 明确声明，系统不需要人为批准个别步骤，只需要最初的批准。

一个关键发现是，多代理方法在速度和成功率方面都优于单代理 LLM 系统。劳动分工的划分允许并行执行侦察和利用任务，批评代理能够捕捉到单一大型代理可能会遗漏的错误。然而，研究人员警告说，当前的 LLM 仍然表现出幻觉和命令注入故障，这可能导致系统停滞或执行意外行动。

## 缓解措施与建议

Unit 42 建议云防御者假设 AI 驱动的攻击将成为常态，并相应地加强环境。具体的缓解措施包括执行最少权限 IAM 策略，不设立常设管理角色，启用云轨迹日志记录和异常检测以监测不寻常的 API 调用序列，以及实施网络分段以限制横向移动。研究人员还建议组织进行红队演习，模拟多代理 AI 攻击者以识别漏洞，抢在对手之前。这项研究没有与 CVE 或产品补丁相关的具体内容；它是关于新兴攻击技术的威胁情报警告。
